Functional Programming

Making simple things complex

January 30, 2016

An hour or so ago, I didn't know anything about functional programming. Keep that in mind as you read the rest of this...it's not as if reading a few articles has made me an expert. Still, having read those articles, I know something about it and it seems like a concept that will perch on one side of my mind as I continue to learn about coding.

So, you're wondering what is this functional programming thing? I'm going to answer that in two parts. First, why should we care about functional programming and, second, how is it done.

The goals of functional programming are safer data and consistently excellent processes. According to this excellent article on the subject, Facebook, Twitter and At&T use functional programing to handle aspects of their massive amounts of data and security.

Functional programming isn't really about a specific programming languages, although there are languages that are more conducive to working this way. Most of the articles I've read use examples from JavaScript and Java and mentioned some languages that are more tailored to functional programming like Scala and Haskell.

There are apparently two fundamental rules of functional programming:

  • No changing data. If you have an array and you need to sort it, store the sorted array in a new spot. No overwriting the original array.
  • Past events can't influence current functions. Each function has to operate every time as if it were the first time it's called. This keeps the operations consistent.
  • Another article I read combines these into one rule: no side effects. The author says, this code "doesn't rely on data outside the current function, and it doesn't change data that exists outside the current function."

Basically, you take every thing you have to do in your code and break it down into its smallest part and you do that part. Every function you create takes in some data as an argument and returns some data. Maybe you have to do a thing more than once but - forgot to mention this - you can't have loops in functional programming! Instead, you'd use recursion to get the job done. (That first article goes through an example in detail.) You might also use built-in functions to accomplish this, like map, reduce, filter, all, any, and find..

I've focused most of my effort so far on understanding how functional programming works, these rules it follows and I can see how breaking tasks into their smallest component operations as necessary can improve processes. Instead of finding a bug in a 1000-line block of code, you look for it in a 3-line block. Looking at the first rule, if you prohibit your code from changing underlying data, you never have to reconstruct your database if something goes wrong.

From the code-maintenance perspective, smaller parts should be easier to understand. I could write a loop to start anywhere and go any number of times through a collection of data. When I use a built-in function like map, that function always works the same way. Anyone who reads my code would know what I'm doing without having to interpret how my mind works.

In general, functional programming seems to have advantages that would apply no matter waht type of program you're creating. It might also help me quit avoiding recursion, which would probably be a good thing.
