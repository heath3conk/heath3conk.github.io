Arrays & Hashes

Organization!

January 3, 2016

In the physical world, I'm a fairly organized person. My books are arranged according to the room they're in, genre and how much I love them (cookbooks in the kitchen, most, most favorite books in my bedroom, so that I could reach them if I were deathly ill). DVDs go by genre and vintage. You get the idea.

In the virtual world, organization is even more essential. You won't be able to find your favorite blog post in your home directory by recognizing the font of its title on the shelf. So you create folders and put things in some kind of logical order.

As you go down into smaller and smaller environments - from a condo to a computer to a Ruby program that's supposed to get something done - the need to keep things in some kind of logical order doesn't go away. With code, you've got several ways to get that done.

You might need to have a single piece of information that you'll use in some way (or several ways). For that, you'd probably have a variable set to a single value. For example, maybe you'd have a variable called first_name. You might have a simple bit of code that asks the user to input their name and you'd store their input in that variable.

Then, suppose you wanted to keep track of everyone who uses your simple bit of code. You'd want to collect their first_name entries into a list you could go back to at a later time. An array is ideal for this purpose. Arrays are just collections of data. In this example, not only would your array maintain a list of users' names, it would also be able to show you the order in which they answered your question. That's because arrays are ordered lists. Every element in an array has a specific position, or index. The index of a given name would indicate whether that person used your name_grabber before or after someone else.

After a few people have come by, your array might look like this: visitors = ["Heather", "Will", "Cathleen"]

Suppose though that you wanted more information than just the user's name? Maybe you want their last_name and birthday too. A hash could keep this information for you. A hash differs from an array in that the data in a hash comes in pairs of a key and a value.

A hash for my information would look like this: {"first_name" => "Heather", "last_name" => "Conklin", "birthday" => "December 16"} The beauty of the hash is that the relationships between the data points is so obvious. Clearly, my birthday is December 16. If the key birthday weren't right there, you might wonder what that date was all about. But that's not a problem with hashes. If you were looking at the information my sister input instead of me, the hash would be: {"first_name" => "Cathleen", "last_name" => "Feduke", "birthday" => "May 7"}

One important note here, the keys in a hash are unique. You can't have two first_name entries in the hash. Values can repeat though, so if my brother's name were "Will Will," his hash would be: {"first_name" => "Will", "last_name" => "Will", "birthday" => "June 1"}

Now, here's where it gets really good (for someone like me who kind of wishes she owned a label-maker). Both arrays and hashes can hold anything. Strings, integers, other arrays or hashes, anything, even combinations of different types of things. So, if I wanted to keep track of all the people who've interacted with my name_grabber code, I might organize it like this: visitors_hash = [{"first_name" => "Heather", "last_name" => "Conklin", "birthday" => "December 16"}, {"first_name" => "Cathleen", "last_name" => "Feduke", "birthday" => "May 7"}, {"first_name" => "Will", "last_name" => "Will", "birthday" => "June 1"}] Notice the different brackets: arrays use [] while hashes use {}.

That is an array whose elements are hashes! I can iterate over the array to find the hash whose first_name key holds a value of "Cathleen" and return the value in her birthday key. Both arrays and hashes have built-in methods for finding information they hold, adding and removing elements, rearranging their contents and otherwise using the data in them (which is really the point).

There are circumstances where it's obvious which of these collections is the "best" way to hold your data and others where you could probably use either. I haven't come across that many circumstances up to this point so I don't have a lot of examples to look at. For the moment, I'm thinking of it this way: If I need to emphasize a relationship between the question I'm asking and the answer I'm getting, then I have a key (the question) and a value (answer) and a hash might be the collection for that task. Otherwise, an array would probably work fine.

In this particular example, you might argue that an array of arrays would be equally efficient, since the three pieces of data I want are all the same and show up in the same order each time. If I wanted to, I could create an array of values for each visitor and build up an array made up of other arrays, like this: visitors_array = [["Heather", "Conklin", "December 16"], ["Cathleen", "Feduke", "May 7"], ["Will", "Will", "June 1"]] Note that the keys are gone as are the "=>" markers. I could still iterate through these arrays, find "Cathleen" and return "May 7" for her birthday, based on that string's position in the array that starts with her name.
